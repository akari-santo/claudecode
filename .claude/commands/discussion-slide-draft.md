# Discussion Slide Draft Skill

## 概要

`project-definition-checker` が出力したチェックリスト診断結果（`project_checklist_YYYYMMDD.md`）と、過去の議事録・外報を統合的に読み込み、次回打ち合わせで**議論を活発にし、より多くの情報を引き出す**ためのスライドドラフト（テキスト構成案）を生成する。

**このスキル自体は `.pptx` ファイルを生成しない。** 出力はテキストベースの「スライドドラフト」であり、実際のスライド化は `consulting-slide` や `concept-slide` スキルに委譲する。

### 設計思想

1. **前提認識の共有が先**: 「ここまでは合意していますよね？」を明示することで、認識齟齬を防ぎ、議論を前に進める
2. **仮説提示で反応を引き出す**: 白紙の質問より「こう考えていますが合っていますか？」の方が情報が出る
3. **論点の可視化で漏れを防ぐ**: 議論すべきアジェンダを事前に見せることで、先方も準備ができる
4. **ASIS→TOBE→HOW→ROADMAP→TECHNICALの流れを踏襲**: 最終提案スライドと同じ構造で議論を進めることで、打ち合わせを重ねるごとに提案スライドが自然に完成に近づく

## トリガー（使用タイミング）

- `project-definition-checker` 実行後、次回打ち合わせのアジェンダ設計を行う場面
- ユーザーが「次の打ち合わせ用のスライドを作りたい」「議論用ドラフトを用意したい」と言った場合
- 外報の NA セクションに「次回お打ち合わせ」のアジェンダが記載されている場合
- 商談の進捗に応じて、仮説ベースの提案方向性を先方に確認したい場合

## 前提条件

以下のファイルが `/mnt/user-data/uploads` に存在すること：

- **必須**: `project_checklist_YYYYMMDD.md`（最新のもの1つ以上）
- **推奨**: 過去の議事録（`mtg_YYMMDD.md` 等）
- **推奨**: 過去の外報

すべてのファイルを読み込み、時系列順に整理してから処理を開始する。

## 思考プロセス（厳守）

### Step 0: 全量インプットの読み込みと時系列整理

1. `/mnt/user-data/uploads` のファイルをすべて読み込む
2. ファイルを以下のカテゴリに分類する：
   - **チェックリスト**: `project_checklist_YYYYMMDD.md`
   - **議事録**: `mtg_YYMMDD.md` 等の打ち合わせ記録
   - **外報**: 外報ファイル
   - **その他資料**: 提案書、見積、技術資料等
3. 日付順に並べ、議論の変遷（どの打ち合わせで何が決まり、何が残ったか）を把握する
4. **最新のチェックリストを「現在地」として使用する**

### Step 1: 情報の3分類

チェックリストと議事録から、全情報を以下の3つに分類する：

| 分類 | 定義 | スライド上の扱い |
|------|------|------------------|
| **✅ 合意済み（Confirmed）** | チェックリストで【済】の項目。具体的数値・事実があり、先方と確認済み | 「前提認識の確認」スライドに明記 |
| **🔶 仮説（Hypothesis）** | チェックリストで【一部】の項目。言及はあるが具体性不足。こちら側で仮説を立てられるもの | 「ご提案イメージ（仮説）」スライドに提示 |
| **❓ 未確認（Open）** | チェックリストで【未】の項目、または仮説すら立てられないもの | 「本日の論点・確認事項」スライドに質問として配置 |

**重要ルール:**
- 合意済みの情報には必ず出典（「第X回MTG（MM/DD）にてご確認」）を付記する
- 仮説には必ず「（仮説）」ラベルを付け、事実と明確に区別する
- 仮説はチェックリストの「不足点・ネクストアクション」と議事録の文脈から論理的に導出する。根拠のない推測は禁止

### Step 2: スライド構成の決定

打ち合わせの回数・フェーズに応じて、ASIS→TOBE→HOW→ROADMAP→TECHNICAL の5カテゴリのうち、**今回の打ち合わせで議論すべき範囲**を決定する。

#### フェーズ判定ロジック

```
IF チェックリストの ASIS が【済】2/2 → ASIS は確認のみ、TOBE を重点議論
IF チェックリストの ASIS・TOBE が概ね【済】 → HOW・Roadmap を重点議論
IF 全カテゴリに【一部】【未】が散在 → 最もブロッカーとなる項目を優先
```

**原則: 1回の打ち合わせでカバーする重点カテゴリは最大2つ。** 詰め込みすぎると議論が浅くなる。ただし、合意済み事項の確認は全カテゴリ横断で簡潔に行う。

### Step 3: スライドドラフトの生成

以下の構造に従って、各スライドのテキストドラフトを生成する。

---

## スライドドラフト構造（テンプレート）

### スライド0: 表紙
```
タイトル: 第X回 お打ち合わせ資料
サブタイトル: [案件名]
日付: YYYY-MM-DD
作成者: [当社名]
```

### スライド1: 本日のアジェンダ
```
Tagline: アジェンダ
Action Title: 本日は[重点カテゴリ]を中心に、[目的]を確認・議論させていただきたい

Body:
1. 前提認識の確認（X分）— これまでの合意事項の振り返り
2. [重点議題1]（X分）— [議題の目的を一文で]
3. [重点議題2]（X分）— [議題の目的を一文で]
4. 次回に向けて（X分）— NA・スケジュールの確認

Footnote: 前回お打ち合わせ: YYYY-MM-DD
```

### スライド2: 前提認識の確認
```
Tagline: 前提認識の確認
Action Title: これまでのお打ち合わせで以下の点を確認・合意いただいており、本日の議論の前提としたい

Body（表形式）:
| カテゴリ | 確認事項 | 内容 | 確認日 |
|---------|---------|------|--------|
| ASIS | [項目名] | [具体的な合意内容] | 第X回（MM/DD） |
| TOBE | [項目名] | [具体的な合意内容] | 第X回（MM/DD） |
| ... | ... | ... | ... |

※ 認識に相違がございましたらご指摘ください

Footnote: 出典: 第1回〜第X回 打ち合わせ議事録
```

**生成ルール:**
- チェックリストのステータスが【済】の項目をすべて列挙する
- 内容はチェックリストの「現状の整理・根拠」セクションから抽出する
- 出典（どの打ち合わせで確認されたか）を必ず明記する
- 情報量が多い場合は、カテゴリごとにスライドを分割する

### スライド3〜N: 重点議題スライド（カテゴリ別）

各重点カテゴリについて、以下の **3層構造** でスライドを生成する。
1カテゴリあたり1〜3枚のスライドとなる。

#### 層1: 現状認識（Confirmed + Hypothesis）
```
Tagline: [カテゴリ名]（例: 現状業務の整理）
Action Title: [合意済み事実を踏まえた現状認識の要約。仮説を含む場合はその旨を明記]

Body:
■ 確認済みの事実
  - [合意済み事項1]（第X回確認）
  - [合意済み事項2]（第X回確認）

■ 弊社の仮説（ご確認いただきたい点）
  - [仮説1]: [根拠となる発言や文脈] → [仮説の内容]
  - [仮説2]: [根拠となる発言や文脈] → [仮説の内容]

Footnote: ■は確認済み事実、仮説は弊社の想定であり、認識相違があればご指摘ください
```

#### 層2: ご提案イメージ（Hypothesis）
```
Tagline: [カテゴリ名] — ご提案の方向性（案）
Action Title: [仮説に基づく提案の方向性を主張として記述]

Body:
（論理構造に応じて比較表、プロセスフロー、概念図などを選択）
- 仮説ベースの提案イメージ
- 複数案がある場合は松竹梅で提示
- 各案のメリット・デメリット・前提条件

※ 本スライドは現時点の仮説に基づく方向性案です

Footnote: 出典: [根拠となるデータソース] | 詳細は次回以降に精緻化
```

#### 層3: 確認・議論ポイント（Open）
```
Tagline: [カテゴリ名] — ご確認事項
Action Title: [提案の精緻化・次フェーズ進行のために、以下をご確認させていただきたい]

Body:
（チェックリストの「不足点・ネクストアクション」から導出）

Q1. [具体的な質問文]
    背景: [なぜこの情報が必要かの簡潔な説明]

Q2. [具体的な質問文]
    背景: [なぜこの情報が必要かの簡潔な説明]

Q3. [具体的な質問文]
    背景: [なぜこの情報が必要かの簡潔な説明]

Footnote: ご回答は概算・方向性レベルで結構です
```

**質問設計のルール:**
- 1スライドあたり最大3問。それ以上は別スライドに分割
- 各質問に「なぜ聞くのか」の背景を1行で添える（先方が回答しやすくなる）
- Yes/No で答えられる質問より、具体的な数値や選択肢を引き出す質問を優先
- チェックリストの「具体的な質問文」をそのまま活用してよい
- SoW作成のブロッカー（チェックリストで優先度高のフラグがある項目）に対応する質問を最優先で配置

### スライドN+1: 次回に向けて（NA整理）
```
Tagline: Next Action
Action Title: 本日のご確認事項を踏まえ、以下のスケジュールで進めさせていただきたい

Body（表形式）:
| 担当 | アクション | 期限 |
|------|-----------|------|
| [先方] | [依頼事項] | YYYY-MM-DD |
| [当社] | [対応事項] | YYYY-MM-DD |

■ 次回お打ち合わせ
  日時: YYYY-MM-DD（仮）
  目的: [次回の論点]
  アジェンダ: [予定する議題]

Footnote: 日程は仮置きです。ご都合を確認のうえ調整させてください
```

## Step 4: ドラフトの品質チェック

生成したドラフトに対して以下を確認する：

### 構造チェック
- [ ] 全スライドが ASIS→TOBE→HOW→ROADMAP→TECHNICAL の流れに沿っているか
- [ ] 合意済み事項が漏れなく「前提認識の確認」に記載されているか
- [ ] 仮説と事実が明確に区別されているか（ラベル付き）
- [ ] チェックリストの【一部】【未】項目がいずれかのスライドでカバーされているか

### 議論設計チェック
- [ ] 1回の打ち合わせの重点カテゴリが最大2つに絞られているか
- [ ] 質問が具体的で、数値や選択肢を引き出せる形になっているか
- [ ] 質問に「なぜ聞くのか」の背景が添えられているか
- [ ] SoW作成のブロッカーに対応する質問が優先的に配置されているか

### コミュニケーションチェック
- [ ] 先方に「準備されている」「ちゃんと聞いてくれている」という印象を与える構成か
- [ ] 仮説が先方の発言を踏まえたものになっているか（的外れな仮説は逆効果）
- [ ] 全体の枚数が適切か（目安: 30分の打ち合わせで6〜10枚、60分で10〜15枚）

## 出力先

- `/mnt/user-data/outputs/slide_draft_YYYYMMDD.md` に出力する
- 日付は次回打ち合わせの予定日とする。未定の場合はドラフト作成日とする

## 処理ルール（厳守事項）

1. **事実と仮説の厳密な分離**: 合意済み事項には必ず出典を付記し、仮説には必ず「（仮説）」ラベルを付ける。この区別が曖昧なスライドは生成しない。
2. **チェックリストとの整合性**: チェックリストの全10項目が、いずれかのスライドで必ずカバーされていること（合意済み→前提確認、仮説→提案イメージ、未確認→確認ポイント）。
3. **議事録の全量読み込み**: 最新のチェックリストだけでなく、過去の議事録・外報もすべて読み込み、議論の変遷を把握したうえでドラフトを作成する。
4. **仮説の根拠明示**: 仮説を立てる際は、どの発言・文脈からその仮説に至ったかを明記する。根拠のない推測は禁止。
5. **質問の優先順位**: SoW作成のブロッカーとなる項目（チェックリストで優先度高フラグあり）に対応する質問を最優先で配置する。
6. **枚数の適正化**: 目安として30分で6〜10枚、60分で10〜15枚。詰め込みすぎない。

## 関連スキルとの連携

| スキル | 関係 | 連携内容 |
|--------|------|---------|
| **project-definition-checker** | 上流（インプット） | チェックリスト診断結果を入力として使用。ステータス・不足点・NAを直接活用 |
| **minute-to-gaiho** | 上流（インプット） | 外報のNA・アジェンダを入力として使用。所感セクションは仮説の根拠として参照 |
| **consulting-slide** | 下流（アウトプット） | 本スキルのドラフトを渡してスライド化を依頼 |
| **concept-slide** | 下流（アウトプット） | 提案の全体像を伝えるスライドの生成を依頼 |

### 運用フロー（全体像）

```
打ち合わせ実施
    ↓
議事録作成（mtg_YYMMDD.md）
    ↓
minute-to-gaiho → 外報作成
    ↓
project-definition-checker → チェックリスト診断
    ↓
★ discussion-slide-draft → スライドドラフト作成（本スキル）
    ↓
consulting-slide / concept-slide → .pptx 生成
    ↓
次回打ち合わせ実施
    ↓
（繰り返し → チェックリストが埋まり、最終的に提案スライドが完成）
```

この循環により、打ち合わせを重ねるごとに ASIS→TOBE→HOW→ROADMAP→TECHNICAL が段階的に埋まり、最終提案スライドが自然に形成されていく。
