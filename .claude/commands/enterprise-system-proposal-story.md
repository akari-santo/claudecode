# Enterprise System Proposal Story Designer

## 概要

大規模基幹システム（レガシーモダナイゼーション、システム統合・連携基盤構築、ERP導入・刷新等）に関する提案資料の**スライド全体のアウトライン・ストーリー構成**を生成するスキル。

**このスキル自体は `.pptx` を生成しない。** 出力は各スライドの論点・問い・推奨図解を含むMarkdownのストーリー設計書（`enterprise_proposal_story_YYYYMMDD.md`）であり、スライド化が必要な場合は別途 `slide-draft-template` → `consulting-slide` を使用する。

### このスキルが解く問い

> 「基幹システムの刷新・統合という大規模投資に対して、経営層・情シス・現場の三者が納得し意思決定できる提案ストーリーをどう組み立てるか？」

基幹システム提案では、AI提案と異なり「技術的な新規性」よりも「業務プロセスの再設計（BPR）」と「移行リスクの制御」が意思決定の中心にある。このスキルは、その意思決定構造に沿ったストーリーを設計する。

### AI提案（proposal-story-designer）との根本的な違い

| 観点 | AI提案 | 基幹システム提案 |
|------|--------|---------------|
| 意思決定の軸 | 「この技術で本当に解けるか？」（技術的不確実性） | 「止まらないか？業務は良くなるか？」（事業継続と業務変革） |
| ステークホルダー | 現場担当 + 技術部門中心 | 経営層 + 情シス + 現場の三層構造 |
| 提案のゴール | PoC承認 | 要件定義フェーズ（またはRFP回答の採択） |
| 投資規模 | 数百万〜数千万円 | 数千万〜数十億円 |
| 最大のリスク | 精度が出ない | 業務が止まる・移行が失敗する |
| ストーリーの核心 | 技術的な実現可能性の証明 | 業務変革の価値 + 移行の安全性の証明 |

### 2つの案件タイプ

本スキルでは、案件を以下の2タイプに分類し、各セクション内の「問い」と力点を切り替える:

| タイプ | 特徴 | ストーリーの力点 |
|--------|------|---------------|
| **モダナイゼーション型** | レガシーシステムの刷新・リプレイス。老朽化・EOL・技術的負債が主な動機 | セクション1でEOLリスクと技術的負債を強調 → セクション4の移行計画が最重要 |
| **システム統合型** | 複数システムの連携基盤構築・データ統合。業務横断の非効率が主な動機 | セクション1でデータサイロと業務断絶を強調 → セクション2のBPR（統合後の業務像）が最重要 |

インプット情報から案件タイプを判定し、各セクションで対応する「問い」を選択する。両方の要素を持つ案件（例: レガシーをリプレイスしつつ複数システムを統合）では、両タイプの問いを併用する。

## 標準フレームワーク（5セクション構成）

基幹システムの意思決定プロセスは以下の5層で構成される:

**「現状を知る → 未来を描く → 投資を正当化する → 安全に移行する → 任せられると証明する」**

---

### セクション1: 【業務の現在地】（As-Is Deep Dive）

**このセクションの役割**: 現状業務の「見えていない非効率」を可視化し、変革の必要性を自分事化させる

基幹システム提案では、顧客自身が「何が問題か」を正確に把握していないことが多い。長年の運用で属人化・サイロ化した業務プロセスを第三者視点で可視化し、「このまま続けるとどうなるか」を突きつけることがセクション1の核心。

#### 共通の問い

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 業務の実態 | 現在の業務プロセスは、どこで人手・紙・Excel・属人的判断に依存しているか？ |
| 組織への影響 | 業務の属人化・データのサイロ化は、経営判断のスピードにどう影響しているか？ |
| 放置コスト | このまま3〜5年間現状を維持した場合のTCOと機会損失はいくらか？ |

#### モダナイゼーション型の追加の問い

| 問いの種別 | 具体的な問い |
|-----------|------------|
| EOLリスク | 現行システムのベンダーサポート終了・OS/ミドルウェアのEOLはいつか？それまでに移行が完了しない場合のリスクは？ |
| 技術的負債の実態 | 長年のカスタマイズ・パッチ適用で、現行システムのソースコードやIF定義はどの程度ブラックボックス化しているか？ |
| 保守体制の脆弱性 | 現行システムを理解している技術者は何名残っているか？ 退職・世代交代のリスクはどの程度か？ |
| 性能・拡張の限界 | 現行基盤はデータ量の増加・ユーザー数の増加・新業務要件にどこまで耐えられるか？ |

#### システム統合型の追加の問い

| 問いの種別 | 具体的な問い |
|-----------|------------|
| データサイロの全体像 | 何系統のシステムに業務データが分散しており、マスタの不整合はどの程度か？ |
| 手動連携の実態 | システム間のデータ連携で、手動入力・CSV受渡し・バッチの遅延はどこで発生しているか？ |
| 業務断絶のコスト | 部門間の情報断絶により、二重入力・確認待ち・手戻りで年間どの程度の工数が失われているか？ |
| 経営情報の分断 | 経営層がリアルタイムで見たい数字（売上・在庫・原価等）のうち、即座に取得できないものは何か？ |

**典型スライド構成**:
- 業務フロー現状分析（As-Is）：主要業務プロセスのフロー図 + ペインポイントのマッピング
- 現行システム構成図：システム間のデータフロー・手動連携・二重入力の可視化
- **[モダナイゼーション型]** EOL / 技術的負債マップ：システム別のリスク評価と期限
- **[モダナイゼーション型]** 保守コスト推移：過去5年の保守費用の上昇トレンドと今後の予測
- **[システム統合型]** データサイロ / マスタ不整合マップ：システム別のデータ分散状況
- **[システム統合型]** 手動連携・業務断絶のコスト試算：工数・リードタイム・ミス率の定量化
- 放置した場合のリスクシナリオ：3〜5年後の最悪ケースを可視化

**このセクションの設計思想**: 基幹システムの顧客は「困っている」と自覚していても、問題の全体像が見えていないケースが多い。特にモダナイゼーション案件ではEOLの「期限」が、統合案件では業務断絶の「累積コスト」が、経営層を動かす最大のトリガーになる。提案者が「見えていなかった問題」を提示することで、洞察力と信頼を示す。

### セクション2: 【変革の設計図】（To-Be Design & BPR）★重点セクション

**このセクションの役割**: 「システムを入れ替える/繋げる」ではなく「業務を再設計する」という提案の価値を示す

基幹システム提案が「ベンダーのパッケージ紹介」や「連携ツールの機能説明」に堕する最大の原因は、To-Beの業務設計なくシステム機能の説明に走ること。このセクションでは「業務がどう変わるか」を先に描き、その結果としてシステムが必要であるという論理構成にする。

#### BPR設計の3層アプローチ

セクション2の設計では、以下の3層を順に組み立てる:

**第1層: 業務プロセスの再設計（業務視点）**
- As-Is業務フロー → ボトルネック・ムダの特定 → To-Be業務フローの描画
- 「なくす」「まとめる」「自動化する」「順序を変える」の4つの改善パターンで整理
- 業務変革により「誰の何が、どう変わるか」を具体的に記述

**第2層: データ・情報の統合設計（データ視点）**
- マスタデータの統合方針（統合 / 名寄せ / コード変換）
- データフロー（どこで生まれ、どこを通り、どこで使われるか）の再設計
- リアルタイム連携 / バッチ連携 / イベント駆動の使い分け

**第3層: システムアーキテクチャ（技術視点）**
- 業務要件から導かれるシステム構成の概念設計
- パッケージ / SaaS / スクラッチの使い分け
- Fit & Gap（標準適用 / カスタマイズ / 運用回避）の方針

#### 共通の問い

| 問いの種別 | 具体的な問い |
|-----------|------------|
| あるべき姿 | 3〜5年後、この業務はどうあるべきか？（業務プロセスの理想形） |
| 変革の範囲 | 今回のスコープでどこまでの業務変革を実現し、何を次フェーズに残すか？ |
| 業務変革の効果 | BPRによる定量的な効果（工数削減・リードタイム短縮・データ一元化等）はどの程度か？ |

#### モダナイゼーション型の追加の問い

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 現行踏襲 vs 再設計 | 現行業務をそのまま新基盤に載せるか、この機会に業務プロセス自体を見直すか？（この判断がプロジェクトの成否を分ける） |
| カスタマイズの清算 | 現行システムで積み上げたカスタマイズのうち、本当に業務上不可欠なものはどれか？ 標準機能に寄せられるものはどれか？ |
| Fit & Gap方針 | パッケージ標準に業務を合わせる（Fit to Standard）のか、業務に合わせてカスタマイズするのか？その判断基準は何か？ |
| 段階的リリース | 全業務を一斉に切り替えるか、業務領域ごとに段階的にリリースするか？ |

#### システム統合型の追加の問い

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 統合後の業務プロセス | 複数システムに跨っていた業務フローが、統合後はどう一本化されるか？ どの手動連携が消え、どの承認フローが変わるか？ |
| マスタ統合の方針 | 顧客マスタ・商品マスタ・組織マスタ等の統合ルール（名寄せ・コード体系・管理主体）はどうするか？ |
| 連携アーキテクチャ | Hub & Spoke / Point-to-Point / イベント駆動のどのパターンを採用し、なぜか？ |
| 段階的統合 | どのシステム間の連携を先に統合し、どれを後回しにするか？ その優先順位の根拠は？ |

**典型スライド構成**:
- **[BPR第1層]** As-Is → To-Be業務フロー対比：改善ポイントをハイライト
- **[BPR第1層]** 業務変革の効果試算：工数・コスト・リードタイムの定量的な改善見込み
- **[BPR第1層]** 業務部門別の変化サマリー：「誰の何が、どう変わるか」一覧
- **[BPR第2層]** データ統合方針：マスタ統合ルール・データフローの再設計
- **[BPR第3層]** To-Beシステム全体構成図：新基盤のアーキテクチャ概要
- **[BPR第3層]** Fit & Gap概要：標準適用 / カスタマイズ / 運用回避の方針マトリクス
- **[モダナイゼーション型]** 現行踏襲 vs 再設計の判断マトリクス：業務領域別の方針
- **[システム統合型]** 連携アーキテクチャ概要：統合パターンと選定理由

**このセクションの設計思想**: 経営層は「業務がどう良くなるか」に関心があり、情シスは「技術的に実現可能か」に関心がある。BPRの3層アプローチにより、業務視点（経営層・現場向け）→ データ視点（情シス・現場向け）→ 技術視点（情シス向け）と、ステークホルダーごとの関心に段階的に応える構成にする。「システムの話」ではなく「業務改革の話」として提案を位置付けることが、コンペにおける最大の差別化になる。

### セクション3: 【投資の論理】（TCO/ROI & Investment Justification）

**このセクションの役割**: 大規模投資に対して、経営判断に必要な「数字の根拠」を提示する

基幹システム提案では、AI提案のような「小さく試して大きくする」アプローチが取りにくい。初期投資が大きいため、経営層には「なぜこの金額を今投じるべきか」を数字で説得する必要がある。

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 投資規模 | 初期投資・ランニングコスト・5年TCOはいくらか？ |
| 投資対効果 | ROI・回収期間（ペイバック期間）はどの程度か？ |
| 比較検討 | 「やらない場合のコスト」と「やった場合のコスト」の差分はどうか？ |
| 段階投資 | フェーズ分けにより初期投資をどこまで抑制でき、各フェーズのROIはどうか？ |

**典型スライド構成**:
- TCO比較：現行維持 vs 新システム導入の5年間コスト比較
- ROI / ペイバック：投資回収シミュレーション
- フェーズ別投資計画：段階的な投資と効果のタイムライン
- リスク調整後の投資評価：楽観・標準・悲観の3シナリオ

**このセクションの設計思想**: 数字の「精度」よりも「論理の透明性」が重要。前提条件を明示し、顧客が自社の数字を当てはめて検証できる構造にする。セクション1の「放置コスト」とセクション2の「BPR効果」を受けて、「投資しない方がリスクが高い」という論理を完成させる。

### セクション4: 【移行の安全保障】（Migration Risk & Transition Plan）★重点セクション

**このセクションの役割**: 「業務が止まらない」「データが消えない」という最大の不安を解消する

基幹システム刷新の最大のリスクは「移行の失敗」であり、この恐怖が意思決定を遅らせる最大の要因。このセクションでは、移行リスクを正面から取り上げ、具体的な制御策を提示することで意思決定の障壁を取り除く。

**移行リスクを「軽く見せる」のは逆効果。** リスクを正面から列挙した上で、それぞれに具体的な対策を紐づけることで、提案者の経験と信頼性を示す。「リスクを隠すベンダー」と「リスクを見通して対策を持っているベンダー」のどちらを選ぶかは明白。

#### 移行計画の4つの柱

セクション4は以下の4つの柱で構成する:

**柱1: 移行方式の選定と根拠**
- ビッグバン / 段階移行 / 並行稼働 / ストラングラーパターンの比較
- 案件特性に応じた最適方式の選定理由
- 各方式のリスク・コスト・期間のトレードオフ

**柱2: データ移行の設計**
- 現行データの棚卸し（量・品質・形式・依存関係）
- クレンジング・変換ルールの策定
- 移行リハーサル計画（何回、いつ、どの範囲で実施するか）
- 移行後のデータ検証（整合性チェック・業務検証）

**柱3: 業務切替の設計**
- 切替時の業務停止時間（ダウンタイム）の最小化策
- ユーザー教育・変更管理（チェンジマネジメント）計画
- 移行期間中の業務代替手段（手動運用フォールバック）
- Go/No-Go判定基準の事前合意

**柱4: リスク管理と切戻し**
- 想定リスクの洗い出しとインパクト評価
- 各リスクに対する予防策と発生時の対応策
- 切戻し（ロールバック）計画：どの時点まで、何時間で戻せるか
- エスカレーションルートと意思決定基準

#### 共通の問い

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 移行方式 | どの移行方式を選び、なぜか？ トレードオフをどう評価したか？ |
| データ移行 | 現行データの品質・量・変換ルールはどう管理するか？ |
| 業務継続 | 切替時の業務停止時間をどう最小化するか？ 移行中の業務代替手段は何か？ |
| 切戻し | 万が一の場合、どの時点まで何時間で切り戻せるか？ |
| 移行リハーサル | 本番移行前に何回リハーサルを行い、どの基準でGo判定するか？ |

#### モダナイゼーション型の追加の問い

| 問いの種別 | 具体的な問い |
|-----------|------------|
| レガシーデータの変換 | 旧システム固有のデータ構造・コード体系を、新システムのデータモデルにどう変換するか？ 変換できない例外データの扱いは？ |
| カスタマイズの移行 | 旧システムのカスタマイズ機能のうち、新システムで代替手段がないものはどれか？ どう対処するか？ |
| ストラングラーパターン | 全面移行が困難な場合、旧システムの一部機能を残しつつ段階的に切り替えるストラングラーパターンは適用可能か？ |
| EOLとの時間競争 | サポート終了までのタイムラインに対して、移行スケジュールに十分なバッファがあるか？ |

#### システム統合型の追加の問い

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 段階的統合の順序 | どのシステム間の連携を先に統合し、全体の統合完了までのロードマップはどうなるか？ |
| 並行運用期間 | 旧連携方式と新連携方式の並行運用期間はどの程度必要か？ データの二重管理をどう制御するか？ |
| IF（インターフェース）の棚卸し | 現行のシステム間IFは何本あり、それぞれの移行難易度・優先度はどうか？ |
| 部門間の合意形成 | 統合により業務プロセスが変わる部門間で、移行タイミング・運用ルールの合意をどう取り付けるか？ |

**典型スライド構成**:
- **[柱1]** 移行方式の比較・選定：各方式のメリット・デメリット・リスク比較表
- **[柱1]** 移行全体スケジュール：フェーズ区分と主要マイルストーン
- **[柱2]** データ移行計画：棚卸し → クレンジング → 変換 → リハーサル → 本番のプロセス
- **[柱2]** データ移行リハーサル計画：回数・スコープ・合否判定基準
- **[柱3]** 業務切替計画：切替タイムラインと業務停止の最小化策
- **[柱3]** チェンジマネジメント計画：ユーザー教育・サポート体制の概要
- **[柱4]** リスク管理マトリクス：想定リスク × 発生確率 × 影響度 × 対策
- **[柱4]** 切戻し計画：Go/No-Go判定基準とロールバック手順
- **[モダナイゼーション型]** レガシーデータ変換計画：変換ルール・例外処理の方針
- **[システム統合型]** 段階的統合ロードマップ：IF単位の統合順序と並行運用計画

**このセクションの設計思想**: 移行の安全性を伝えるには「大丈夫です」という言葉ではなく「こういうリスクがあり、こう対処します」という具体性が必要。4つの柱（方式選定・データ移行・業務切替・リスク管理）を網羅することで、提案者が移行の全体像を見通していることを示す。特にモダナイゼーション案件では「レガシーデータの変換」と「EOLとの時間競争」が、統合案件では「段階的統合の順序」と「部門間の合意形成」が、成否を分ける論点になる。

---

### セクション5: 【実行体制の証明】（Capability & Governance）

**このセクションの役割**: 「この会社・チームに任せて大丈夫」という信頼を勝ち取る

大規模プロジェクトでは「何をやるか」と同じくらい「誰がやるか」が重要。提案者の実行力・ガバナンス体制・類似実績を示し、発注の最終判断を後押しする。

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 体制 | プロジェクト体制（PMO・開発・業務コンサル）はどう構成するか？ |
| ガバナンス | ステアリングコミッティ・進捗管理・品質管理の仕組みはどうなるか？ |
| 実績 | 類似業界・類似規模のプロジェクト実績はあるか？ |
| ベンダー選定根拠 | 推奨する製品・技術スタックを選んだ根拠は何か？ |

**典型スライド構成**:
- プロジェクト体制図：顧客側・提案者側の役割と責任
- ガバナンス体制：意思決定プロセス・報告ライン・エスカレーション
- 推進スケジュール：全体マイルストーン（要件定義〜本番稼働〜安定化）
- 類似実績・ケーススタディ：業界・規模・課題の類似性を強調
- 技術スタック / ベンダー比較：選定理由と評価軸の提示

## 処理手順

### Step 0: 案件タイプの判定

インプット情報を読み込んだ段階で、以下の判定を行う:

| 判定基準 | モダナイゼーション型 | システム統合型 | 複合型 |
|---------|-------------------|-------------|--------|
| 主なキーワード | EOL, 刷新, リプレイス, 老朽化, 技術的負債, マイグレーション | 統合, 連携, データ基盤, マスタ統合, サイロ, 一元化 | 両方の要素が混在 |
| 現行システム | 単一の大型レガシーが対象 | 複数システムが乱立 | レガシー刷新 + 他システムとの統合 |
| 主な動機 | サポート終了・保守コスト・技術者不足 | 業務横断の非効率・データ分断 | 両方 |

**複合型の場合**: 両タイプの問いを併用し、セクション1でそれぞれのリスク/課題を並列で提示する。

### Step 1: インプットの読み込みと解析

以下のファイルを `/mnt/user-data/uploads` から読み込む（存在するものすべて）:

- 議事録 (`mtg_YYMMDD.md` 等)
- 外報 (`gaiho_YYMMDD.md` 等)
- 先方からのRFP / RFI / 提案依頼書 (`.pdf`, `.pptx`, `.md` 等)
- 既存スライド / 過去提案書 (`.pptx` 等)
- チェックリスト診断結果 (`project_checklist_*.md`)
- 現行システムの資料（構成図、マニュアル等）

読み込んだ情報から以下を抽出・推定する:

```
■ 案件タイプの判定結果
  - モダナイゼーション型 / システム統合型 / 複合型
  - 判定の根拠（どの情報からそう判断したか）

■ 先方の意思決定構造
  - 経営層が気にしていること（投資規模、ROI、経営戦略との整合性）
  - 情シス部門が気にしていること（技術的実現性、運用負荷、ベンダーロックイン）
  - 現場部門が気にしていること（業務の変化、使い勝手、移行時の負担）
  - 最終意思決定者は誰か（CxO、情シス部長、事業部長等）

■ 案件の特性
  - 対象システム（ERP / CRM / SCM / レガシー / 複合）
  - 現行システムの状況（パッケージ名、バージョン、EOL状況、カスタマイズ状況）
  - 対象業務の範囲（会計 / 販売 / 生産 / 購買 / 人事 / 物流等）
  - 業界特有の規制・コンプライアンス要件

■ 提案の位置付け
  - RFP回答 / コンペ / 既存関係からの提案 / 情報提供フェーズ
  - 競合ベンダーの有無と差別化ポイント
  - 先方が期待する次のステップ（要件定義 / PoC / 比較検討）
```

### Step 2: フレームワークのカスタマイズ判断

5セクション構成をベースに、案件タイプとインプット情報に応じてセクションの重みづけ・順序を調整する。

**案件タイプ別のデフォルト重みづけ**:

| セクション | モダナイゼーション型 | システム統合型 |
|-----------|-------------------|-------------|
| S1: 業務の現在地 | ★★★（EOLリスク・技術的負債を強調） | ★★★（データサイロ・業務断絶を強調） |
| S2: 変革の設計図 | ★★（現行踏襲 vs 再設計の判断が焦点） | ★★★★（統合後の業務像・BPR効果が最重要） |
| S3: 投資の論理 | ★★ | ★★ |
| S4: 移行の安全保障 | ★★★★（移行計画が最重要） | ★★★（段階的統合のロードマップ） |
| S5: 実行体制の証明 | ★★ | ★★ |

**追加のカスタマイズ判断**:

| 状況 | 調整方針 |
|------|---------|
| 先方が「現行の問題は明確」と言っている | セクション1を簡略化し、セクション2（To-Be/BPR）を手厚く |
| RFPで詳細な要件が提示されている | セクション1・2を簡略化し、セクション4・5を手厚く |
| 経営層プレゼンが主目的 | セクション3（TCO/ROI）を手厚く、技術詳細を控えめに |
| 情シス向けの技術提案が主目的 | セクション2のアーキテクチャとセクション4の移行計画を手厚く |
| 競合コンペである | セクション5の差別化（実績・体制・ベンダー比較）を強化 |

### Step 3: スライド一覧の生成

各スライドについて以下の情報を生成する:

| 要素 | 説明 |
|-----|------|
| スライド番号・タイトル | 1枚のスライドが扱う内容の一言要約 |
| 所属セクション | 5セクションのどこに属するか |
| 案件タイプ対応 | [共通] / [モダナイゼーション型] / [システム統合型] |
| 対象ステークホルダー | このスライドが主に誰に刺さるか（経営層 / 情シス / 現場） |
| このスライドで答える問い | 聴衆が「で、何がいいたいの？」と思う問いを1つ設定 |
| Action Titleの仮案 | So what? の主張（1〜2行） |
| 推奨図解タイプ | slide-draft-templateの図解タイプ名から選択 |
| インプットから使える情報 | 議事録・RFP等のどの記述が使えるか |
| 不足している情報 | スライドを完成させるために追加で必要な情報 |

### Step 4: ストーリーライン検証

生成したスライド一覧に対して、以下の視点で自己チェックを行う:

**論理の一貫性チェック**:
- [ ] セクション1（As-Is）を読んで「確かに今のままではまずい」と危機感を持てるか
- [ ] セクション2（To-Be/BPR）を読んで「こう変わるなら価値がある」と思えるか
- [ ] セクション3（投資）を読んで「この金額は妥当だ」と納得できるか
- [ ] セクション4（移行）を読んで「止まらないし安全だ」と安心できるか
- [ ] セクション5（体制）を読んで「この会社に任せよう」と決断できるか
- [ ] セクション1→2→3の流れで「放置コスト > 投資コスト」の論理が成立しているか
- [ ] セクション2→4の流れで「BPRの理想像が移行計画で具体的に実現される」と繋がっているか

**ステークホルダー別チェック**:
- [ ] 経営層が15分で読んでも意思決定に必要な情報が揃っているか
- [ ] 情シスが技術的に突っ込んでも答えられる深さがあるか
- [ ] 現場担当が「業務が変わる」ことに対する不安が解消されるか

**案件タイプ整合性チェック**:
- [ ] モダナイゼーション型の場合：EOLリスクと移行計画が十分な深さで扱われているか
- [ ] システム統合型の場合：データサイロの解消とBPR効果が具体的に描かれているか

### Step 5: 不足情報のサマリー

ストーリー設計上「このスライドを作るには先方への確認が必要」な情報を一覧化する。基幹システム提案では特に以下が不足しがち:

- 現行システムの具体的な構成・バージョン・カスタマイズ状況
- 業務別のトランザクション量・ユーザー数
- 移行対象データの量と品質
- 予算感・投資枠の上限
- 他社提案の有無・評価基準
- **[モダナイゼーション型]** 現行システムのソースコード・IF定義の状態
- **[システム統合型]** 現行のシステム間IF一覧と連携方式の詳細

## 処理上の注意事項

1. **インプットがない場合**: ユーザーに業種・対象システム・現行環境・主要課題を質問してから処理を開始する。案件タイプ（モダナイゼーション / 統合 / 複合）の確認も行う
2. **RFP回答の場合**: RFPの評価項目・配点を読み取り、配点の高い項目に対応するセクションを手厚くする
3. **スライド枚数の目安**: 5セクションで合計15〜25枚（表紙・目次・エグゼクティブサマリーを含む）。BPR（セクション2）と移行計画（セクション4）は他セクションより枚数が多くなる傾向がある
4. **SoW部分の扱い**: セクション5の「推進スケジュール」は概要のみ記載し、要件定義フェーズの詳細は `sow-generator` で別途生成する旨を明記する
5. **プレースホルダーの活用**: 確認が必要な数値・固有名詞は `[要確認]` で明示する
6. **ステークホルダー別の深さ**: 同じスライドでも「経営層向けの読み」と「情シス向けの読み」がある場合は、Appendixスライドの推奨を付記する
7. **競合差別化の意識**: コンペの場合、各セクションで「なぜ他社ではなく当社か」の差別化メッセージを埋め込むポイントを明示する
8. **案件タイプの混在**: 複合型の場合、各スライドにどちらのタイプの「問い」を使っているかを明記し、ストーリー全体の一貫性を保つ
