# Estimate Review Request — エンジニア向け期間見積もりレビュー依頼文生成

## 概要

営業がエンジニアに開発期間の見積もり（またはレビュー）を依頼する際、
「期間だけ教えて」と丸投げすると、情報不足のまま安全マージンを大きく取った回答が返ってきがちである。

このスキルは、エンジニアが精度の高い見積もりを出すために必要な情報を
**営業側が先に整理して渡す**ためのSlackメッセージを生成する。

---

## 設計思想

### エンジニアが見積もりに必要な4軸

1. **何を作るか（スコープと要件）** — 機能一覧、優先度、入出力仕様、非機能要件、「やらないこと」の明確化。PoCなら検証仮説と成功判定基準。
2. **どう作るか（技術構成と不確実性）** — 技術スタック、外部連携、既存システムとのインテグレーション、未知領域の大きさ。
3. **データ・環境** — データ提供元・形式・量、前処理の複雑さ、環境準備状況。
4. **外部依存・待ち時間** — 先方レビューのリードタイム、データ提供時期、アカウント/権限発行の所要日数、社内レビュープロセス。

加えて **体制（誰がやるか）** — アサイン候補、掛け持ち状況、営業側の希望納期。

### テンプレート設計の原則

- 全部埋まらなくてもよい。空欄のまま送って「ここが分からない」と伝えること自体が価値。
- 冒頭の「確認したいこと」で **最短〜現実的〜安全マージン込みの3パターン** を明示的に依頼する。レビュアーが最初に「何を求められているか」を把握できる。
- Slackにそのまま貼れるMarkdown形式で出力する。
- 絵文字・表は使わない。構造化は箇条書きのネストで表現する。

---

## フェーズ判定

ユーザーの入力から以下のいずれかを判定する。明示がなければ確認する。

- **PoC** — 「検証」「PoC」「技術検証」「プロトタイプ」「仮説検証」など
- **本開発** — 「本開発」「システム開発」「実装」「プロダクション」「本番」「リリース」など

## 実行手順

### Step 1: インプット確認

ユーザーから以下のいずれかが提供されることを想定する：
- 議事録、外報、提案資料、チェックリスト診断結果
- 口頭での案件概要説明
- 「この案件の見積もり依頼を出したい」という指示のみ

提供された情報から、テンプレートの各項目を可能な限り埋める。

### Step 2: フェーズ判定

PoCか本開発かを判定する。明示がなければユーザーに確認する。
両方必要な場合は2つ出力する。

### Step 3: セクション記述

提供された情報をもとに、各セクションの `{自由記述}` 部分を埋める。

記述ルール：
- 各セクションは自由記述。箇条書きでも散文でも、情報量に応じて最適な形で書く
- セクション名の下の補足行（_イタリック_）は「こういう観点で書くとよい」というガイド。1項目1バレットで全部埋める必要はない
- 情報の階層化が必要な場合は箇条書きのネストで表現する。表（テーブル）は使わない
- 絵文字は使わない
- 分かっている情報 → 具体的に記載
- 分からない情報 → `（未確認）` と記載して残す。空欄で残すこと自体に価値がある
- 営業側の仮説がある場合 → `〜という認識（要確認）` の形で記載

### Step 4: 出力

Slackにそのまま貼れる形式で出力する。
message_compose_v1 ツールを使い、kind="other" で出力する。

Slack記法ルール：
- ボールド: `*text*`（Slack準拠、そのまま）
- イタリック: `_text_`（Slack準拠、そのまま）
- 箇条書き: `• `（中黒ドット U+2022）を使う。`- ` はSlackで箇条書きとして描画されないため使わない
- 子項目のインデント: `    • `（半角スペース4つ + `• `）で表現する
- 区切り線: `——`（全角ダッシュ2つ）を使う。`---` はSlackで水平線にならないため使わない
- 表（テーブル）は使わない。情報の階層化は箇条書きのネストで表現する
- 絵文字は使わない

案件名・回答期限が不明な場合は `〇〇` のままプレースホルダーとして残す。

## PoCと本開発の差分ポイント

テンプレート選択時に以下の差分を意識する：

- **セクション1** — PoCは「検証ゴール（仮説・成功判定）」、本開発は「スコープ・要件（機能一覧・非機能要件）」
- **技術構成** — PoCは学習/チューニングの有無を重視、本開発はインフラ/デプロイ・PoCからの引き継ぎを追加
- **データ・環境** — PoCは検証環境の準備状況、本開発はデータ移行・本番環境・テストデータ
- **外部依存** — PoCはFB・データ提供・権限発行、本開発は+リリース判定基準・QAプロセス
- **確認事項** — PoCは3項目（見積もり可否・3パターン・リスク）、本開発は4項目（+フェーズ分割提案）

## 補足：営業→エンジニアのコミュニケーション設計意図

このテンプレートは単なる「情報の穴埋めシート」ではなく、以下のコミュニケーション効果を狙っている：

1. **情報の非対称性を減らす** — 営業が持っている顧客文脈をエンジニアに構造的に伝える
2. **見積もりの精度を上げる** — 「期間だけ教えて」の丸投げを防ぎ、根拠ある見積もりを引き出す
3. **3パターン回答を引き出す** — 最短/現実的/安全の幅があることで、先方への提示の柔軟性が生まれる
4. **不足の可視化** — 空欄が残っていること自体が「ここはまだ分からない」という有用な情報になる
