## 概要
議事録（Meeting Minutes）から、情報量を維持しつつ要点を構造化した「外報（外部向け報告書）」を生成します。ファシリテーションとコンサルティングの両視点に基づき、単なる要約ではなく「解釈」と「次のアクション」に重きを置きます。

## トリガー（使用タイミング）
- 議事録が更新された場合に、その日付の外報を作成する
- `mtg_yymmdd.md` などのファイルが更新または新規作成された場合
- 商談終了直後のメモが入力された場合

# 外報作成（コンサルティング営業用）

## 思考プロセス
1. **情報抽出**: 入力されたテキストから、質疑応答、フィードバック、決定事項、期限、数値、固有名詞を漏れなく抽出する。
2. **構造化**: 指定の「外報」フォーマットに当てはめる。特に「概要」セクションでは、インデントを用いて議論の構造（問いと答え）を可視化する。
3. **解釈の付加**: 事実に基づき、コンサルタントとしての「課題背景」「リスク」「ROIへの期待」を考察し「所感」にまとめる。
4. **アクションの具体化**: 「誰が・何を・いつまでに」を明確にしたNAリストを作成する。
5. **欠落情報のチェック**: もし「期限」や「担当者」が不明な場合は、推測せず `(要確認)` と付記した上で、後述の `PENDING_ACTIONS.md` に記載する。

## 出力フォーマット（厳守）
# 外報

## 概要（事実に基づく状況把握）
- 先方や当社からの質問事項を起点に、回答や議論内容をインデントによって構造化してまとめる
- **内容説明の FB**：先方からのフィードバック／質疑応答／修正依頼
- **稟議フロー・見積デッドライン等スケジュール**：承認プロセス、必要書類、期限（YYYY-MM-DD）

## 所感（状況・発言に対する自分なりの解釈）
- 事実から読み取れる課題・背景
- リスクや依存関係の洞察
- 価値・ROI への期待度合い
- 不確かな情報には `(要確認)` を付す

## NA（Next Action：解釈から導き出される行動）
- **先方**：
  - 依頼事項／準備物と担当・期限
- **当社（燈）**：
  - 担当・タスク・期限
- **次回お打ち合わせ**：YYYY-MM-DD または TBD  
  - **目的**：何をヒアリング／合意するのか  
  - **アジェンダ**：目的から導かれる議題を箇条書き

## 表示ルール
- 日付：YYYY-MM-DD、未確定は TBD
- 引用：重要なニュアンスは 「 」 で抜粋
- 情報量：落とさず、かつ端的に

## 出力の2形式と出力順序

外報は以下の **2形式** を **両方** 出力する。

### 形式1：Slack貼り付け用テキスト（会話内にそのまま出力）
- 会話のレスポンス本文として、コードブロックや装飾なしにそのままテキストで出力する
- ユーザーがそのままコピーして Slack に貼り付けられる状態にする
- 変換ルール：
  - `## 見出し` → `■ 見出し` に置換（Slack は Markdown 見出しを解釈しないため）
  - タイトル行は `━━━━━━━━━━━━━━━━━━` の区切り線で囲む
  - `**太字**` → `*太字*`（Slack のボールド記法）
  - 箇条書き `-` はそのまま維持。インデント（半角スペース）も維持し、段落構造を崩さない
  - 日付・引用（「 」）・`(要確認)` 等の表記はそのまま維持
  - Markdown のリンク `[text](url)` → `text (url)` に展開

### 形式2：Markdown版（.md ファイルとして出力）
- 「出力フォーマット（厳守）」セクションに従った従来通りの Markdown 形式
- `.md` ファイルとして `/mnt/user-data/outputs/` に保存し、`present_files` で共有する
- ファイル名例：`gaiho_250314_サンプル製造.md`

### 出力順序
1. まず **形式1（Slack用テキスト）** を会話内に出力する
2. 次に **形式2（Markdown版）** を `.md` ファイルとして作成・共有する

## 特記事項
もし、外報を作成する上で「ユーザーに確認しないとNAが確定できない」等のボトルネックが発生した場合は、プロジェクトルートの `PENDING_ACTIONS.md` にその内容を追記し、ユーザーに判断を仰ぐこと。
