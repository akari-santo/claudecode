# Proposal Story Designer Skill

## 概要

AIシステム検証開発に関する提案資料の**スライド全体のアウトライン・ストーリー構成**を生成するスキル。

**このスキル自体は `.pptx` を生成しない。** 出力は各スライドの論点・問い・推奨図解を含むMarkdownのストーリー設計書（`proposal_story_YYYYMMDD.md`）であり、スライド化が必要な場合は別途 `slide-draft-template` → `consulting-slide` を使用する。

### このスキルが解く問い

> 「インプット情報をもとに、先方が納得し意思決定できる提案ストーリーをどう組み立てるか？」

顧客の意思決定プロセスに沿った論理展開（Why → What → How を定義 → How を証明）を設計し、各スライドが「何を伝えるべきか」を明確化する。

---

## 標準フレームワーク（4セクション構成）

### セクション1: 【問題の発見】（Why & Gap）

**このセクションの役割**: 顧客の痛みを「自分事化」させる

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 現状の解像度 | 現場で「誰が」「いつ」「何に」最も苦労しているか？ |
| 負の連鎖 | その問題が放置されることで組織全体にどんな悪影響が出るか？ |
| 阻害要因 | なぜ既存の仕組みや従来のIT化では解決できなかったのか？ |

**典型スライド構成**:
- 業務フロー現状分析（As-Is）→ ボトルネックの特定
- 放置した場合のコスト・リスクの定量化
- 理想状態（To-Be）との差分＝AIが必要な理由

---

### セクション2: 【問題の選定】（Prioritization & ROI）

**このセクションの役割**: 投資の妥当性を経営層に納得させる

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 全体像における位置付け | 業務全体のバリューチェーンの中で、今回扱う領域はどこか？ |
| 選定の妥当性 | 多くの課題の中で、なぜ「ここ」を最優先で解くべきなのか？ |
| 投資価値 | この問題を解いた際のROIは後続フェーズへの投資を正当化できるか？ |

**典型スライド構成**:
- 課題の全体マップ（ロジカルツリー or 課題一覧）
- 優先順位マトリクス（インパクト × 実現性）→ 今回の対象を明示
- 導入ロードマップ（Quick Win → 中長期拡張）

---

### セクション3: 【問題設定の定義】（Input/Output & Logic）

**このセクションの役割**: 開発側とユーザー側の「期待値のズレ」をなくす

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 境界条件 | どこからどこまでをAIが担い、どこを人間が担うか？ |
| データの性質 | 入力データの揺らぎや構造をどう捉えるか？ |
| 成功の定義 | 何をもって「解けた」とするか？（評価指標/KPIの定義） |

**典型スライド構成**:
- I/Oデータ定義・データモデル整理
- データフロー図 or システムアーキテクチャ（概念レベル）
- 検証指標の定義（定量ゴール）

---

### セクション4: 【解法】（How & Feasibility）

**このセクションの役割**: 実行のリアリティと信頼を勝ち取る

| 問いの種別 | 具体的な問い |
|-----------|------------|
| 技術的アプローチ | 設定された問題を解くための最適な技術スタックは何か？ |
| 不確実性の制御 | AI特有の「精度リスク」に対してどう検証するか？ |
| 実行の確実性 | なぜ「我々（提案者）」がこの問題を解けると言えるのか？ |

**典型スライド構成**:
- AI/技術アプローチの説明（なぜこの手法か）
- PoC設計の概要（→ 詳細は `sow-generator-poc` に委譲）
- 推進体制・マイルストーン・実績

## 処理手順

### Step 1: インプットの読み込みと解析

以下のファイルを `/mnt/user-data/uploads` から読み込む（存在するものすべて）:

- 議事録 (`mtg_YYMMDD.md` 等)
- 外報 (`gaiho_YYMMDD.md` 等)
- 先方からの提案依頼書・RFP (`.pdf`, `.pptx`, `.md` 等)
- 既存スライド (`.pptx` 等)
- チェックリスト診断結果 (`project_checklist_*.md`)

読み込んだ情報から以下を抽出・推定する:

```
■ 先方の期待感・要件
  - 先方が解決したいと言っている問題（表面）
  - 先方が本当に不安に思っていること（裏面）
  - 意思決定者が気にするであろうポイント（ROI、リスク、実績）

■ 案件の業界・業務文脈
  - 業界（製造/建設/金融/小売/etc.）
  - 対象業務（どのような業務プロセスか）
  - データの特性（非定型文書/画像/数値/etc.）

■ AI開発の技術文脈
  - 解こうとしている問題の分類（LLM/画像解析/数値予測/etc.）
  - データ取得・整備の難易度
  - 類似事例・業界標準のパターン
```

### Step 2: ストーリーの骨子設計

インプット解析結果をもとに、**顧客の意思決定ロジック**に沿ったストーリーの骨子を設計する。

**設計の優先原則**:
1. **先方の言語で語る**: 技術用語ではなく、先方の業界・業務の言葉を使う
2. **痛みの深さに応じてセクション1に重みを置く**: 先方の問題意識が明確なほど、セクション1を手厚くする
3. **決裁者と現場担当者を意識した2層構造**: セクション2のROI説明は決裁者向け、セクション3の定義は現場向け
4. **不確実性を隠さない**: AI特有の精度リスクはセクション4で正直に提示し、PoCで制御する姿勢を示す

**カスタマイズ判断**:
- インプットに「すでに問題の優先順位は決まっている」とある → セクション2を簡略化
- 先方がデータ不足を懸念している → セクション3のデータ定義を手厚く
- 技術的な信頼性への疑問がある → セクション4の実績・体制を強化
- 「まずPoC」と合意済みである → セクション4でSoWへの橋渡しを明示

### Step 3: スライド一覧の生成

各スライドについて以下の情報を生成する:

| 要素 | 説明 |
|-----|------|
| スライド番号・タイトル | 1枚のスライドが扱う内容の一言要約 |
| 所属セクション | 4セクションのどこに属するか |
| このスライドで答える問い | 聴衆が「で、何がいいたいの？」と思う問いを1つ設定 |
| Action Titleの仮案 | So what? の主張（1〜2行） |
| 推奨図解タイプ | slide-draft-templateの図解タイプ名から選択 |
| インプットから使える情報 | 議事録・RFP等のどの記述が使えるか |
| 不足している情報 | スライドを完成させるために追加で必要な情報 |

### Step 4: ストーリーライン検証

生成したスライド一覧に対して、以下の視点で自己チェックを行う:

- [ ] セクション1（Why）を読んで「確かに困っている」と共感できるか
- [ ] セクション2（Prioritization）を読んで「なぜここを解くべきか」が納得できるか
- [ ] セクション3（Definition）を読んで「何をどこまでやるか」が明確になるか
- [ ] セクション4（How）を読んで「この会社なら任せられる」と思えるか
- [ ] 全体を通じて「なぜやるか → 何からやるか → 何を解くか → どう解くか」の流れが一貫しているか

### Step 5: 不足情報のサマリー

ストーリー設計上「このスライドを作るには先方への確認が必要」な情報を一覧化する。

## 処理上の注意事項

1. **インプットがない場合**: ユーザーに業種・業務・AI化したい内容を質問してから処理を開始する
2. **先方要件が不明確な場合**: 議事録から「先方が繰り返し言及している言葉」を重点的に拾い上げる
3. **スライド枚数の目安**: セクション1〜4で合計10〜16枚（表紙・目次・エグゼクティブサマリーを含む）。インプットの情報量に応じて増減する
4. **SoW部分の扱い**: セクション4の「PoC設計」スライドは概要のみ記載し、詳細は `sow-generator-poc` で別途生成する旨を明記する
5. **プレースホルダーの活用**: 確認が必要な数値・固有名詞は `[要確認]` で明示する
