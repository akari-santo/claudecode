# Slide Draft Template Skill

## 概要

ユーザーの断片的な情報（メモ、箇条書き、口頭説明、議事録の一部など）を受け取り、`consulting-slide` スキルが最高品質のpptxを生成できる形式のテキストドラフトに変換する。

**このスキル自体は `.pptx` ファイルを生成しない。** 出力はテキストベースの「スライドドラフト」であり、スライド化は `consulting-slide` スキルに委譲する。

### なぜこのスキルが必要か

`consulting-slide` スキルはドラフトの解析（Step 0）から始まるが、入力ドラフトの品質にアウトプット品質が大きく左右される。特に以下の問題が起きやすい:

- Action Title が事実説明で終わり、主張になっていない
- 図解タイプが不明確で、スキル側の自動判定がブレる
- 強調ポイントが曖昧で、Accent Color が散らばる
- 出典が欠落し、Footnote がプレースホルダーだらけになる

このスキルは、これらの問題を**ドラフト段階で防ぐ**ための型を提供する。

## トリガー（使用タイミング）

- ユーザーがスライドの構成案・下書きを求めた場合
- 断片的なメモや箇条書きを「スライド用に整理して」と依頼された場合
- `discussion-slide-draft` の出力を `consulting-slide` 向けに再整形する場合
- 「Action Title を考えて」「So what? を整理して」等、スライドの論理構造設計を求められた場合
- 複数スライドのストーリーライン設計を求められた場合

## 出力フォーマット（厳守）

1スライドにつき、以下のブロックを1つ生成する。複数スライドの場合はブロックを連番で並べる。

````markdown
## スライド [番号]: [内容を一言で]

### Tagline
[スライドが扱う領域。数単語。]

### Action Title
[結論 = So what?。「〜により、〜すべきである」「〜の結果、〜が必要」等の主張を含む1〜3行。]

### 図解タイプ
[以下から1つ選択]
- comparison / process / hierarchy / kpi / table
- matrix / timeline / roadmap / waterfall
- flowchart / architecture / venn / before-after / grouping
- data-model / swimlane / process-deep-dive

### Body
[図解タイプに応じた構造化データ。後述のサブフォーマットに従う。]

### Accent
[Body 内で最も伝えたい1点を明記。]

### Footnotes
[出典・注釈。不明な場合は「出典: [要確認]」]
````

## Body のサブフォーマット（図解タイプ別）

### comparison（比較）

```
#### 比較軸
- 列ヘッダー: [項目名] | [比較A] | [比較B]

#### データ行
- [行1ラベル]: [値A] → [値B]
- [行2ラベル]: [値A] → [値B]
（3〜6行が目安）
```

### process（プロセスフロー）

```
#### ステップ
1. [ステップ名]: [補足説明]
2. [ステップ名]: [補足説明]
3. [ステップ名]: [補足説明]
（3〜5ステップが目安）

#### 強調ステップ
- Step [番号]
```

### hierarchy（階層・ロジックツリー）

```
#### 構造
- [頂点]
  - [分岐1]
    - [要素1-1]
    - [要素1-2]
  - [分岐2]
    - [要素2-1]

#### MECEの切り口
- [分類軸を明記]
```

### kpi（数値コールアウト）

```
#### KPI一覧（2〜4個）
1. [数値]: [ラベル] / [補足]
2. [数値]: [ラベル] / [補足]

#### 強調KPI
- [番号]番目
```

### table（構造化テーブル）

```
#### テーブル構造
- 列: [列1名] | [列2名] | [列3名]

#### データ
| [列1] | [列2] | [列3] |
|--------|--------|--------|
| ...    | ...    | ...    |

#### 強調セル
- [行, 列]の組み合わせ
```

### matrix（2x2マトリクス）

用途: BCGマトリクス、SWOT、優先度マトリクス等。2軸で4象限に分類する。

```
#### 軸定義
- X軸: [左ラベル] ←→ [右ラベル]
- Y軸: [下ラベル] ←→ [上ラベル]

#### 象限の内容
- 右上（[象限名]）: [要素1, 要素2, ...]
- 左上（[象限名]）: [要素1, 要素2, ...]
- 右下（[象限名]）: [要素1, 要素2, ...]
- 左下（[象限名]）: [要素1, 要素2, ...]

#### 強調象限
- [右上 / 左上 / 右下 / 左下]
```

記入例:
```
#### 軸定義
- X軸: 市場成長率 低 ←→ 高
- Y軸: 市場シェア 低 ←→ 高

#### 象限の内容
- 右上（Star）: プロダクトA, プロダクトC
- 左上（Cash Cow）: プロダクトB
- 右下（Question Mark）: プロダクトD, プロダクトE
- 左下（Dog）: プロダクトF

#### 強調象限
- 右下（Question Mark → 投資判断が必要）
```

### timeline（マイルストーンタイムライン）

用途: プロジェクトの時系列イベント、主要マイルストーンを時間軸上に配置。1つのストリームの時系列表現。

```
#### 時間軸
- 範囲: [開始] 〜 [終了]
- 単位: [月 / 四半期 / 年]

#### マイルストーン
1. [時期]: [イベント名] — [補足]
2. [時期]: [イベント名] — [補足]
3. [時期]: [イベント名] — [補足]
（3〜8個が目安）

#### 強調マイルストーン
- [番号]番目
```

記入例:
```
#### 時間軸
- 範囲: 2026/Q1 〜 2027/Q2
- 単位: 四半期

#### マイルストーン
1. 2026/Q1: PoC完了 — 主要3業務での検証結果レビュー
2. 2026/Q2: 要件定義FIX — システム要件・業務要件の最終合意
3. 2026/Q3: 開発Phase1完了 — コアモジュールリリース
4. 2026/Q4: UAT — ユーザー受入テスト・フィードバック反映
5. 2027/Q1: 本番稼働 — 段階的ロールアウト開始
6. 2027/Q2: 定着化完了 — KPI達成確認

#### 強調マイルストーン
- 5番目（本番稼働）
```

### roadmap（フェーズ×領域ロードマップ）

用途: 複数ワークストリームの並行進行を「横軸=時間、縦軸=領域」のマトリクスで表現。timeline との違いは、複数の並行レーンを持つ点。

```
#### 時間軸
- フェーズ: [Phase1名(期間)] | [Phase2名(期間)] | [Phase3名(期間)]

#### ワークストリーム（縦軸の各レーン）
- [レーン1名]:
  - Phase1: [タスク・内容]
  - Phase2: [タスク・内容]
  - Phase3: [タスク・内容]
- [レーン2名]:
  - Phase1: [タスク・内容]
  - Phase2: [タスク・内容]
  - Phase3: [タスク・内容]
（2〜5レーンが目安）

#### 強調セル
- [レーン名] × [Phase番号]
```

記入例:
```
#### 時間軸
- フェーズ: Phase1 要件定義(3ヶ月) | Phase2 開発(6ヶ月) | Phase3 展開(3ヶ月)

#### ワークストリーム
- 業務設計:
  - Phase1: 業務フロー可視化・To-Be設計
  - Phase2: 業務マニュアル作成・テスト
  - Phase3: 現場トレーニング・定着化
- システム開発:
  - Phase1: アーキテクチャ設計・技術選定
  - Phase2: コアモジュール開発・API連携
  - Phase3: 性能チューニング・本番移行
- 組織・チェンジマネジメント:
  - Phase1: ステークホルダー分析・推進体制構築
  - Phase2: パイロット部署でのトライアル
  - Phase3: 全社展開・効果測定

#### 強調セル
- システム開発 × Phase2
```

### flowchart（条件分岐フロー）

用途: 判断基準・意思決定ロジック・業務分岐を可視化。process との違いは、条件分岐（Yes/No）がある点。

```
#### フロー
1. [開始ノード]
2. 判断: [条件文?]
   - Yes → [処理A]
   - No → [処理B]
3. [処理A] → [次のステップ]
4. 判断: [条件文?]
   - Yes → [処理C]
   - No → [処理D]
5. [終了ノード]

#### 強調ノード
- [ステップ番号]（最も重要な判断ポイント）
```

記入例:
```
#### フロー
1. 案件受領
2. 判断: 予算規模 1,000万円以上?
   - Yes → 提案書作成（フル版）
   - No → 簡易見積対応
3. 提案書作成 → 社内レビュー
4. 判断: 競合あり?
   - Yes → 差別化戦略策定 → 最終提案
   - No → 標準提案 → 最終提案
5. クロージング

#### 強調ノード
- 2（予算規模の判断が案件対応の分岐点）
```

### architecture（システム構成図・レイヤー図）

用途: システムアーキテクチャ、技術スタック、レイヤー構造の可視化。

```
#### レイヤー構成（上から下の順）
- [レイヤー1名]: [構成要素1, 構成要素2, ...]
- [レイヤー2名]: [構成要素1, 構成要素2, ...]
- [レイヤー3名]: [構成要素1, 構成要素2, ...]

#### 接続・データフロー
- [要素A] → [要素B]: [接続の説明]
- [要素C] ↔ [要素D]: [双方向の説明]

#### 強調レイヤー or 要素
- [レイヤー名 or 要素名]
```

記入例:
```
#### レイヤー構成
- プレゼンテーション層: Webアプリ, モバイルアプリ, 管理画面
- API層: APIゲートウェイ, 認証サービス
- ビジネスロジック層: 仕様書比較エンジン, AI推論モジュール, ワークフロー管理
- データ層: PostgreSQL, Elasticsearch, S3

#### 接続・データフロー
- Webアプリ → APIゲートウェイ: REST API
- AI推論モジュール → Elasticsearch: ベクトル検索
- ワークフロー管理 → S3: ドキュメント格納

#### 強調レイヤー or 要素
- ビジネスロジック層（差別化の源泉）
```

### venn（ベン図）

用途: 2〜3つの集合の重複・共通領域を可視化。共通価値や重複課題の表現に。

```
#### 集合（2〜3個）
- [集合A名]: [固有要素1, 固有要素2, ...]
- [集合B名]: [固有要素1, 固有要素2, ...]
- [集合C名]（任意）: [固有要素1, 固有要素2, ...]

#### 重複領域
- [A ∩ B]: [共通要素]
- [A ∩ C]: [共通要素]（3集合の場合）
- [B ∩ C]: [共通要素]（3集合の場合）
- [A ∩ B ∩ C]: [全共通要素]（3集合の場合）

#### 強調領域
- [重複領域名]（← 最も伝えたいポイント）
```

記入例:
```
#### 集合
- 自社の強み: AI技術力, 業界知見, アジャイル開発体制
- 顧客ニーズ: 業務効率化, コスト削減, データ活用
- 競合が弱い領域: 業界特化AI, 現場密着型支援

#### 重複領域
- 自社 ∩ 顧客: 業務効率化AI
- 自社 ∩ 競合が弱い: 業界特化×アジャイル
- 自社 ∩ 顧客 ∩ 競合が弱い: 業界特化AIによる現場密着型業務改革

#### 強調領域
- 3集合の交差（スイートスポット）
```

### before-after（ビフォーアフター）

用途: 変革・改善の前後を左右対比で見せる。comparison との違いは、明確な「変化の方向性（→）」を持ち、中央に変革要因を配置する点。

```
#### Before（左側）
- [現状の状態1]
- [現状の状態2]
- [現状の状態3]

#### 変革要因（中央）
- [何が変化を起こすか: 施策名・導入するもの]

#### After（右側）
- [改善後の状態1]
- [改善後の状態2]
- [改善後の状態3]

#### 強調ペア
- [Before/Afterどの行の変化が最もインパクトがあるか]
```

記入例:
```
#### Before
- 仕様書照合: 担当者が手作業で目視比較（2日/件）
- 変更検知: 過去版との差分を手動チェック（漏れ率15%）
- ナレッジ: 個人の記憶と経験に依存

#### 変革要因
- AI仕様書比較・検索システムの導入

#### After
- 仕様書照合: AIが自動比較し差分をハイライト（10分/件）
- 変更検知: 全文自動スキャンで差分検出（漏れ率0.5%）
- ナレッジ: 検索可能なデータベースとして組織に蓄積

#### 強調ペア
- 仕様書照合の工数変化（2日→10分）
```

### grouping（カテゴリ別グルーピング）

用途: 複数の要素をカテゴリで分類してカード状に配置。MECE な分類整理や、施策のカテゴリ分けに。

```
#### グループ（2〜4カテゴリ）
- [カテゴリ1名]:
  - [要素1-1]: [補足]
  - [要素1-2]: [補足]
- [カテゴリ2名]:
  - [要素2-1]: [補足]
  - [要素2-2]: [補足]
- [カテゴリ3名]:
  - [要素3-1]: [補足]
  - [要素3-2]: [補足]

#### 分類の切り口
- [何を基準にグルーピングしたか]

#### 強調カテゴリ or 要素
- [カテゴリ名 or 要素名]
```

記入例:
```
#### グループ
- Quick Win（3ヶ月以内）:
  - RPA導入: 定型帳票の自動生成（投資: 500万円）
  - チャットボット: 社内問合せの一次対応（投資: 300万円）
- 中期施策（6〜12ヶ月）:
  - CRM刷新: 営業活動の可視化・予測（投資: 8,000万円）
  - データ基盤: 全社データレイク構築（投資: 1.2億円）
- 長期構想（1年以上）:
  - AI経営ダッシュボード: リアルタイム意思決定支援（投資: 2億円）

#### 分類の切り口
- 実現までのリードタイム（短期 / 中期 / 長期）

#### 強調カテゴリ or 要素
- Quick Win（早期の成功体験が全社展開の推進力になる）
```

### data-model（データモデル / 概念モデル図）

用途: 業務で扱うデータや情報の概念を整理し、システムやAIアルゴリズムで取り扱えるデータモデルとして可視化する。データの粒度、エンティティ間の関係性（1:1, 1:N, N:N）、各データがどこに存在するかの共通認識を形成する。要件定義フェーズやAI設計の初期段階で特に有効。

```
#### エンティティ一覧
- [エンティティ1名]:
  - 粒度: [1レコード = 何を表すか。例: 「1仕様書」「1部品」]
  - 主要属性: [属性1, 属性2, 属性3, ...]
  - データソース: [どこに保管/記載されているか。例: 「基幹システム」「Excel台帳」「紙図面」]
- [エンティティ2名]:
  - 粒度: [...]
  - 主要属性: [...]
  - データソース: [...]
- [エンティティ3名]:
  - 粒度: [...]
  - 主要属性: [...]
  - データソース: [...]

#### リレーション（関係性）
- [エンティティA] → [エンティティB]: [関係性(1:1 / 1:N / N:N)] — [関係の意味]
- [エンティティB] → [エンティティC]: [関係性] — [関係の意味]

#### 補足（任意）
- データ品質の懸念: [例: 「紙図面のOCR精度」「命名規則の不統一」]
- スコープ外: [今回のモデルに含めないデータ]

#### 強調エンティティ or リレーション
- [最も設計上重要なエンティティ or 関係性]
```

記入例:
```
#### エンティティ一覧
- 仕様書:
  - 粒度: 1仕様書バージョン（同一仕様書の版違いは別レコード）
  - 主要属性: 仕様書ID, 版数, 発行日, 発行元, ステータス, 本文テキスト
  - データソース: 文書管理システム（PDF）+ 一部紙図面
- 部品:
  - 粒度: 1部品番号
  - 主要属性: 部品番号, 部品名, カテゴリ, サプライヤー, 図面番号
  - データソース: 基幹システム（SAP）
- 要求仕様項目:
  - 粒度: 仕様書内の1要求事項（1文〜1段落単位）
  - 主要属性: 項目ID, 要求内容, 対象部品, 判定基準, 適合状況
  - データソース: 仕様書から抽出（AI解析対象）

#### リレーション
- 仕様書 → 要求仕様項目: 1:N — 1つの仕様書に複数の要求事項が含まれる
- 要求仕様項目 → 部品: N:N — 1つの要求が複数部品に適用され、1部品に複数要求が紐づく
- 仕様書 → 仕様書: 1:N — 親仕様書と子仕様書（参照関係）

#### 補足
- データ品質の懸念: 旧仕様書は紙スキャンPDFのためOCR精度に依存
- スコープ外: 試験結果データ（Phase2で追加予定）

#### 強調エンティティ or リレーション
- 要求仕様項目（AI解析のコア対象であり、粒度定義が精度に直結する）
```

### swimlane（スイムレーン図）

用途: 業務プロセスやシステム処理フローを、担当者・システム・工程ごとのレーン（横帯）に分けて記述し、各ステップの実行主体とI/O（入力・出力）を明確にする。現状業務フロー（As-Is）、システム導入後フロー（To-Be）、アルゴリズム処理フローのいずれにも使える。

```
#### レーン定義（実行主体）
- [レーン1名]: [役割の説明。例: 「設計担当者」「AI処理エンジン」「承認者」]
- [レーン2名]: [役割の説明]
- [レーン3名]: [役割の説明]
（2〜5レーンが目安）

#### フローステップ
1. [レーン名] | [アクション名]
   - Input: [このステップへの入力]
   - Output: [このステップの出力]
```

記入例:
```
#### 対象プロセス
- プロセス名: 仕様書セクション間のセマンティック類似度算出
- 全体フローにおける位置: Step ④（全6工程中）

#### Input データ
- 新版仕様書の構造化テキスト:
  - 形式: JSON（セクション単位に分割済み）
  - 粒度: 1セクション = 仕様書内の1章・1節
  - 特性・注意点: セクション長が10文字〜5,000文字と大きくばらつく。表・数式を含むセクションはテキスト化時に情報欠落のリスクあり
  - サンプルイメージ: {"section_id": "3.2.1", "title": "耐熱性要件", "body": "本部品は連続使用温度150℃以上に耐えること...", "has_table": true}
- 旧版仕様書の構造化テキスト:
  - 形式: 同上
  - 粒度: 同上
  - 特性・注意点: 旧版はOCR経由のため、数値誤認識（0→O、1→l等）が一定割合で混入

#### Output データ
- セクション間類似度マトリクス:
  - 形式: CSV（新版セクションID × 旧版セクションID の類似度スコア）
  - 粒度: セクションペア単位
  - 期待される品質水準: 人手判定との一致率90%以上（類似/非類似の2値判定時）
  - サンプルイメージ: |新3.2.1|旧3.2.1: 0.92|旧3.2.2: 0.45|旧3.3.1: 0.12|

#### 技術的不確実性
- セクション長のばらつき: 極端に短いセクション（10文字以下）ではembeddingの意味表現が不安定になる
- OCR由来のノイズ: 旧版の数値誤認識が類似度スコアを過小評価させるリスク
- 表・数式の情報欠落: テキスト化できない構造情報がセクションの意味を変える可能性

#### 採用する技術手法・アプローチ
- 手法: 多言語対応Sentence Transformerによるセクションembedding + コサイン類似度
- 選定理由: 日本語技術文書に対応し、セクション単位の意味比較に適している。キーワードマッチ（BM25等）では同義語・言い換えに対応できないため、セマンティック手法を優先
- 代替案: BM25 + LLM Re-ranking（精度は高いが推論コストが高く、全ペア比較には不向き）
- 検証方針: PoCで人手アノテーション50ペアと照合し、一致率90%を閾値とする

#### 強調ポイント
- セクション長のばらつきへの対処（短文セクションの扱い方がシステム全体の精度を左右する）
```

## 思考プロセス

### Step 1: 入力の解析

ユーザーの入力を読み取り、以下を抽出・推定する:

1. **何枚のスライドが必要か** — 論点が複数ある場合は分割を提案
2. **各スライドのテーマ** — 大きく何について語るスライドか
3. **データと主張の分離** — 事実（数値・状況）と主張（判断・提言）を分ける

### Step 2: Action Title の構築

これがドラフト品質の核心。以下のプロセスで構築する:

1. ユーザーの入力から「事実」を抽出する
2. 「その事実から言えること（So what?）」を推論する
3. 「だから何をすべきか（Implication）」を主張として言語化する

**チェック:** Action Title が以下のいずれかのパターンを満たすか確認する:
- 「〜により、〜すべきである」
- 「〜の結果、〜が不可欠である」
- 「〜を踏まえ、〜へ移行する必要がある」
- 「〜が示すように、〜を優先すべきである」

単なる事実の羅列になっていたら書き直す。

### Step 3: 図解タイプの選択

伝えたいメッセージの構造から最適な図解タイプを選ぶ:

| メッセージの構造 | 図解タイプ |
|----------------|-----------|
| AとBの違い・数値差を見せたい | comparison |
| 時系列の手順・ステップを示したい | process |
| 全体→部分の構造を分解したい | hierarchy |
| インパクトのある数字で説得したい | kpi |
| 多項目を整理して一覧したい | table |
| 2軸で分類・ポジショニングしたい | matrix |
| 時間軸上のマイルストーンを並べたい | timeline |
| 複数領域の並行進行をフェーズで見せたい | roadmap |
| 条件分岐のある判断ロジックを示したい | flowchart |
| システム構成やレイヤー構造を見せたい | architecture |
| 集合の重複・共通領域を可視化したい | venn |
| 変革の前後をインパクトで対比したい | before-after |
| 要素をカテゴリ別に分類整理したい | grouping |
| データの粒度・関係性を共通認識にしたい | data-model |
| 業務/システム処理を担当別レーンで整理したい | swimlane |
| 1工程のI/O・技術手法を深掘り解説したい | process-deep-dive |

**迷ったときの判断フロー:**
1. 「比較」が軸か？ → comparison（数値差） / before-after（変革前後） / matrix（2軸分類）
2. 「時間」が軸か？ → process（手順） / timeline（マイルストーン） / roadmap（並行ストリーム）
3. 「構造」が軸か？ → hierarchy（上下分解） / architecture（レイヤー） / grouping（カテゴリ分類）
4. 「データ」が主題か？ → data-model（概念整理） / process-deep-dive（1工程のI/O深掘り）
5. 「フロー」を見せたいか？ → process（一直線） / flowchart（分岐あり） / swimlane（担当別レーン）
6. 「数字」が主役か？ → kpi
7. 「重なり」を見せたいか？ → venn
8. それ以外 → table（最も汎用）

### Step 4: Accent の決定

「このスライドで、聴衆が1つだけ覚えて帰るとしたら何か？」を自問し、その1点を Accent に指定する。複数指定は禁止。

### Step 5: 出力と確認

ドラフトを出力したら、ユーザーに以下を確認する:
- Action Title の主張が意図に合っているか
- 図解タイプの選択に違和感はないか
- 強調ポイント（Accent）はこれでよいか

確認後、ユーザーが「スライド化して」と言えば `consulting-slide` スキルにそのまま渡せる。

## 複数スライドのストーリーライン設計

5枚以上のスライドセットを求められた場合、個別スライドの前にストーリーライン（全体構成）を先に提示する:

```markdown
## ストーリーライン

1. [スライド1テーマ] — [このスライドの役割: 問題提起 / 現状分析 / etc.]
2. [スライド2テーマ] — [役割]
3. [スライド3テーマ] — [役割]
...

### 全体のロジック流
[起点] → [展開] → [結論]
```

ストーリーラインをユーザーに確認してから、個別スライドのドラフトに進む。

## 依存関係

- 出力したドラフトは `consulting-slide` スキルの入力としてそのまま使用可能
- `discussion-slide-draft` スキルの出力を入力として受け取る場合がある
- `concept-slide` スキル向けのドラフトとして使う場合は、図解タイプを `concept` と記載し、concept-slide スキルに委譲する旨を明記する
