# SoW Generator (PoC Edition) Skill

## 概要

提案資料・企画書・議事録・チェックリスト診断結果等のインプットを読み込み、**PoC（技術検証）プロジェクトの SoW（Statement of Work / 作業明細書）** を6枚構成のスライドドラフト（Markdown）として自動生成する。

**このスキル自体は `.pptx` ファイルを生成しない。** 出力はテキストベースの「スライドドラフト」であり、実際のスライド化は `consulting-slide` スキルに委譲する。

### 要件定義版との違い

| 観点 | 要件定義版（sow-generator） | PoC版（本スキル） |
|------|--------------------------|-----------------|
| 目的 | 何を作るかを定義する | 技術的に実現可能かを検証する |
| Slide 1 | 導入後の業務フロー | SoW記載事項の全体像 |
| Slide 2 | 成果物・完了要件 | 検証項目（何を検証するか） |
| Slide 3 | 要件定義対象 | 評価内容（どう評価するか） |
| Slide 4 | 要件定義書の記載項目 | 作業分担（準備・評価・報告） |
| Slide 5 | 作業分担 | 検証サイクル・報告体制 |
| Slide 6 | 工程表 | 工程表（Phase構成） |
| スコープの変動性 | 低い（合意後は固定） | 高い（アジャイル的に変動） |
| 成果物 | 要件定義書 | 検証報告書 + 次フェーズ判断材料 |

### 設計思想

1. **PoCは「やってみないと分からない」**: だからこそ、事前に検証項目・評価基準・成功判定を明確にし、ゴールポストを固定する
2. **合意前提の明示**: 「本内容を合意したことを前提としてプロジェクトを遂行する」ことを冒頭で明記し、スコープクリープを防ぐ
3. **定量/定性の明確化**: 各検証項目の評価が定量か定性かを明示し、定性評価には顧客協力が必要であることを事前合意する
4. **Phase構造**: PoC（Phase 1）→ 本開発（Phase 2以降）の段階構成を前提とし、次フェーズへの接続を意識する

## トリガー（使用タイミング）

- ユーザーが「PoC SoW」「検証SoW」「技術検証のSoW」等と言った場合
- AI・ML関連の技術検証プロジェクトのSoWを作成する場面
- 提案が通り、まずPoCフェーズから開始する際の作業範囲定義が必要な場面
- 「検証項目を整理したい」「評価基準を決めたい」「PoCのスコープを定義したい」等の言及がある場合

## 前提条件

以下のファイルが `/mnt/user-data/uploads` に1つ以上存在すること：

- **推奨**: 提案資料・企画書（.pptx, .pdf, .md 等）
- **推奨**: 議事録（`mtg_YYMMDD.md` 等）
- **推奨**: `project_checklist_YYYYMMDD.md`（チェックリスト診断結果）
- **推奨**: 既存のSoWテンプレートや過去のSoW

すべてのファイルを読み込み、内容を把握してから処理を開始する。

## PoC SoW 6枚構成（厳守）

SoWは **必ず以下の6枚** で構成する。

---

### Slide 1: SoWにおいて記載する事項（全体像）

**目的**: SoWに何が書かれているかの全体像を示し、「本内容を合意したことを前提としてプロジェクトを遂行する」という合意形成の起点とする

**記載項目**:

```
Tagline: SoW とは？
Action Title: SoW において記載する事項

Body:
  冒頭メッセージ（赤字強調）:
    両者同じゴールを目指してプロジェクトを進行するため、以下を記載。
    検証内容について、本内容を合意したことを前提としてプロジェクトを遂行する。

  表形式:
  | 項目               | 想定される記載事項                                       | 備考                                                         |
  |--------------------|---------------------------------------------------------|--------------------------------------------------------------|
  | 成果物・完了要件     | 検証期間の成果物としてご納品する対象物                      |                                                              |
  | 検証項目           | 検証の対象とする AI 機能と、各項目の対象データ              |                                                              |
  | 作業分担           | 検証に付随して発生が想定される両者の作業事項              |                                                              |
  | 評価内容           | 各検証項目に対する評価内容                                | 定量評価の難しい項目については、定性評価を行う（ご協力が必要）ことを明記 |
  | 検証サイクル       | 検証を実施する上での、ご報告及び、日々のご連絡体制          |                                                              |
  | 作業内容・工程表   | プロジェクト期間における双方の作業内容と、想定スケジュール  | データ用意の想定期限や、定性評価でのご協力にかかる期間も明記       |
```

**抽出ルール**:
- 上記6項目は**デフォルトで固定**。インプットに関係なく常にこの6行を出力する
- 「備考」列のみ、インプットの内容に応じてカスタマイズする
- 冒頭の合意形成メッセージは**必ず含める**（プロジェクトの前提条件として極めて重要）

---

### Slide 2: 検証項目

**目的**: 何を検証するかを明確にし、各検証項目の対象データ・取扱量を定義する

**記載項目**:

```
Tagline: 検証項目
Action Title: 本プロジェクト期間で、以下の項目を検証

Body（表形式）:
  | No. | 検証項目             | 対象データ                                   | 想定取扱量 |
  |-----|---------------------|---------------------------------------------|-----------|
  | 1   | [検証項目名]         | [対象データの説明]                            | 約XX件     |
  |     | （機能単位で記載）    | 絞り込み条件も記載:                           |           |
  |     |                      | ・拡張子                                     |           |
  |     |                      | ・年度                                       |           |
  |     |                      | ・規模（面積, 階数, データサイズ, ...）        |           |
  |     |                      | ・発注者, 建物種, 構造物, ... など             |           |
  |     |                      |                                               |           |
  |     |                      | （スコープアウトするデータも                   |           |
  |     |                      |  リスクに繋がりそうなものは明記）               |           |
  | 2   | [検証項目名2]        | [対象データ2]                                 | 約XX件     |
  | 3   | [検証項目名3]        | [対象データ3]                                 | 約XX件     |
  | -   |                      | 本 Phase では実施せず、次フェイズ以降での実施を検討 |         |
```

**抽出ルール**:
- インプットから**AI機能・検証対象の機能**を「機能」単位で抽出し、検証項目として列挙する
- 各検証項目の**対象データ**を具体的に記載する：
  - データの種類・形式
  - 絞り込み条件（拡張子、年度、規模、カテゴリ等）
  - **スコープアウトするデータ**：明示的にスコープアウトすることが決まったデータ、およびリスクに繋がりそうなものを先方が不安がらせない範囲でここに記載する
- **想定取扱量**は簡潔に「約XX件」とし、詳細は「対象データ」列に記載する
- **最終行に「本 Phase では実施せず、次フェイズ以降での実施を検討」行を必ず設ける**（Phase 2 への接続を示す）
- SoWより前の提案資料部分で具体的な手法説明が必要な場合は、そちらを参照する旨を記載する

---

### Slide 3: 評価内容

**目的**: 各検証項目をどのように評価するかを定義し、成功基準を事前に合意する

**記載項目**:

```
Tagline: 評価内容
Action Title: 各検証項目に対し、以下の内容を評価

Body（表形式）:
  | No. | 検証項目         | 評価内容                                             | 定量 or 定性 |
  |-----|-----------------|------------------------------------------------------|-------------|
  | 1   | [検証項目名]     | [評価方法を具体的に記述]                              | 定量 / 定性  |
  |     | （検証項目）      | 定量評価: どういう定量指標なのか                       |             |
  |     |                  | （% なら、分子/分母）を書く。                         |             |
  |     |                  | 特殊な指標を使うなら、別ページ参照と書く               |             |
  |     |                  |                                                       |             |
  |     |                  | 定性評価で且つ、先方の手を借りるのであれば、            |             |
  |     |                  | その旨をちゃんと分かるように表現                       |             |
  | 2   | [検証項目名2]    | [評価方法2]                                           | 定量 / 定性  |
  | 3   | [検証項目名3]    | [評価方法3]                                           | 定量 / 定性  |
  | -   |                  | 本 Phase では実施せず、次フェイズでの実施を検討         |             |
```

**抽出ルール**:
- Slide 2 の各検証項目に対応する行を生成する（No. は Slide 2 と一致させる）
- **定量評価**の場合：
  - 具体的な定量指標を記述する（精度 %, F1スコア, 処理時間 等）
  - `%` 指標の場合は**分子と分母を明示**する（例: 「正解数 / 全件数」）
  - 特殊な指標（独自KPI等）を使う場合は「別ページ参照」と書く
- **定性評価**の場合：
  - 評価方法を具体的に記述する（例: 「AI出力をExcelシートに記載し、先方にFB頂く」）
  - **先方の協力が必要な場合はその旨を明記する**（PoCの成否に直結するため重要）
- 「定量 or 定性」列は**それ以外の値は書かない**（「定量」「定性」のいずれかのみ）
- **最終行に「本 Phase では実施せず」行を設ける**（Slide 2 と同様）

### Slide 4: 作業分担

**目的**: PoC検証に付随する双方の作業を明確にし、顧客側の協力範囲を事前合意する

**記載項目**:

```
Tagline: 作業分担
Action Title: 検証に付随して、以下の作業が双方に発生することを想定

Body:
  冒頭メッセージ（赤字強調）:
    技術検証は燈のワークとなるが、プロジェクト成功のため貴社にも積極的な関与を頂きたい。

  表形式:
  | カテゴリ | 項目                       | 貴社                                           | 燈                                     |
  |---------|---------------------------|------------------------------------------------|----------------------------------------|
  | 準備    | データの準備               | データの取りまとめ・送付                         | -                                      |
  |         |                           | ※ [送付方法] 経由で、[形式] 形式で送付          |                                        |
  |         | 想定入力/正解データの用意   | Excel で「入力」「正解」の対応表を              | -                                      |
  |         | [LLM・図面？の際は必須]    | 作成 [形式を指定]                               |                                        |
  |         |                           | （[テーマ]分 [数量を指定]）                      |                                        |
  | 評価    | 定性評価                   | ・出力結果の確認                                | ・AI 出力を Excel シートで              |
  |         |                           | ・AI 出力を記載した Excel シートへの              |   取りまとめて送付 [LLMの場合]         |
  |         |                           |   FB 記入 [LLMの場合]                            |                                        |
  | 報告    | 日々の報告                 | [ツール名] 上で双方議論                          | -                                      |
  |         | 報告書の作成               | -                                               | 手法説明・数ケースに対する深い考察      |
  |         |                           |                                                  | ・改善策の考案などを整理して記載        |
  |         | 報告会ミーティング          | [XX様] にご参加頂くのが望ましい。               | 報告準備/当日のファシリテーション       |
  |         |                           | 最終報告には [YY様] にご参加頂く。               |                                        |
```

**抽出ルール**:
- 作業分担は**3カテゴリ固定**で整理する：「準備」「評価」「報告」
- **準備カテゴリ**:
  - 「データの準備」: データの送付方法・形式を具体的に指定する（インプットに記載があれば反映）
  - 「想定入力/正解データの用意」: LLM系・図面系のプロジェクトでは**必須**。Excelでの対応表作成を依頼する形式。テーマ数・数量を具体的に指定する
  - データ形式・送付方法が不明の場合は `[XXX経由で、XXX形式で送付]` とプレースホルダーを入れる
- **評価カテゴリ**:
  - 定性評価が含まれる場合、**顧客側にFB記入を依頼する作業**を必ず含める
  - LLMの場合は「AI出力をExcelシートに記載してFB記入」が典型パターン
- **報告カテゴリ**:
  - 日々の連絡ツール（メール、Teams、Slack等）を指定する
  - 報告会ミーティングの参加者を明記する（特に**最終報告の参加者**は重要）
  - 不明の場合は `[XX様]`、`[YY様]` 等のプレースホルダーを入れる
- **冒頭メッセージ**（「技術検証は燈のワークとなるが…」）は**必ず含める**

### Slide 5: 検証サイクル・報告体制

**目的**: 検証の進め方（改善サイクル）と報告・連絡の体制を定義する

**記載項目**:

```
Tagline: 検証サイクル
Action Title: 検証を実施する上での、ご報告及び日々のご連絡体制

Body:
  ■ 検証サイクル（反復プロセス）
    定量評価 → 改善 → 定量評価 → 改善 → ... （繰り返し）
    + 定性評価（並行して実施）

  ■ 報告体制
    ・定例報告: [頻度]（例: 隔週）で進捗・中間結果を報告
    ・日々の連絡: [ツール名] で随時やり取り
    ・最終報告: プロジェクト終盤に最終報告会を設定

  ■ 成果物
    ・検証報告書（手法説明、評価結果、改善策、次フェーズへの提言）
    ・検証に使用したデータ・コード（必要に応じて）
```

**抽出ルール**:
- 検証サイクルは**「定量評価 → 改善」の反復** + **定性評価の並行実施** を基本パターンとする
- インプットに具体的な検証サイクルの記述があれば反映する
- 報告体制（頻度・ツール・参加者）はインプットから抽出。不明の場合はプレースホルダーを使用
- 成果物はデフォルトで「検証報告書」を含める

### Slide 6: 工程表（ガントチャート）

**目的**: プロジェクトの全体スケジュールをPhase構成で可視化し、マイルストーンを合意する

**記載項目**:

```
Tagline: 工程表
Action Title: 本プロジェクトは、以下の工程で進行

Body:
  冒頭メッセージ:
    技術検証/アジャイルの性質より一定の変動可能性は認めた上で、以下の工程を想定。

  ガントチャート形式:
  |                    | PJ期間（ご契約） |                    |           | Phase2.以降 |
  |                    | [M-1月]  | [M月]         | [M+1月]       | [M+2月]    | [M+3月〜]  |
  |                    | (準備)   | W1  W2  W3  W4 | W1  W2  W3  W4 |           |            |
  |--------------------|----------|-----------------|-----------------|-----------|------------|
  | 共通業務           |          | KO  定例 報告 定例 | 報告 定例 報告 定例 | 最終報告 | KO  最終報告 |
  |   連絡手段の整備   | ████     |                 |                 |           |            |
  | 準備               |          |                 |                 |           |            |
  |   正解データ準備   | ████     |                 |                 |           |            |
  |   環境整備         |   ██████ |                 |                 |           |            |
  |     検証環境整備   |   ████   |                 |                 |           |            |
  |     DB変換         |     ████ |                 |                 |           |            |
  |     前処理         |       ██ |                 |                 |           |            |
  | 検証               |          |                 |                 |           |            |
  |   No.1 [検証項目1] |          | ████ 改善 ████ 改善 | ████ 改善  |           |            |
  |   No.2 [検証項目2] |          |                 | [適宜配置]      |           |            |
  |                    |          | 定性評価 ─────────────────────── |           |            |
  | 次期取組           |          |          次期取組の方針について議論 |           |            |
  | 成果物             |          |                 |    報告資料     | 実施      |            |

  凡例: ■燈担当  ■貴社担当  ■双方

  マイルストーン:
    キックオフお打ち合わせ [MM/DD]
    最終報告お打ち合わせ [MM/DD] ※ [本部長さま] 参加を想定
```

**抽出ルール**:
- **Phase構成**を前提とする：Phase 1（PoC）→ Phase 2（本開発）以降
- 工程は以下の5ブロックで構成する：
  1. **共通業務**: 連絡手段の整備、定例・報告のリズム
  2. **準備**: 正解データ準備、検証環境整備（DB変換、前処理等）
  3. **検証**: Slide 2 の各検証項目を行として展開。「定量評価 → 改善」の反復サイクルを表現
  4. **次期取組**: 次フェーズの方針議論のタイミング
  5. **成果物**: 報告資料の作成・提出タイミング
- **「技術検証/アジャイルの性質より一定の変動可能性は認めた上で」** の但し書きを冒頭に**必ず含める**
- **マイルストーン**（KO日、最終報告日）を明記する。最終報告には上位者（本部長等）の参加を想定する旨を添える
- 定例・報告の頻度はインプットから抽出。デフォルトは**隔週定例 + 隔週報告の交互**
- **定性評価**は検証期間全体にわたる横長バーで表現する（定量評価と並行して実施）
- 貴社担当・燈担当・双方のバーを色分けで区別する旨を凡例に明記する
- インプットに期間情報がなければ以下のデフォルト配分を使用：
  - **準備期間**: 0.5〜1ヶ月（データ準備・環境整備）
  - **検証期間**: 1.5〜2ヶ月（評価→改善の反復）
  - **報告・クロージング**: 0.5ヶ月（報告資料作成・最終報告会）

## 思考プロセス（厳守）

### Step 0: 全量インプットの読み込み

1. `/mnt/user-data/uploads` のファイルをすべて確認・読み込む
2. ファイルを以下のカテゴリに分類する：
   - **提案資料・企画書**: SoWの主要インプット（AI機能・検証手法の記載を特に注視）
   - **議事録**: 合意事項・検証対象の詳細
   - **チェックリスト**: 診断結果（あれば）
   - **既存SoW**: 参考テンプレート（あれば）
3. インプットの全体像を把握する。特に以下を重点的に抽出：
   - **AIの対象機能は何か**（画像認識、LLM、数値予測 等）
   - **対象データの種類と量**
   - **顧客が期待している成果・精度レベル**
   - **プロジェクト期間**

### Step 1: プロジェクトタイプの判定

インプットから、検証の性質を判定する：

| タイプ | 特徴 | 作業分担への影響 |
|--------|------|----------------|
| **LLM系** | テキスト入出力、プロンプト設計 | 「入力」「正解」対応表の作成が必須。定性評価のFBがExcelシートで必要 |
| **画像/図面系** | 画像認識、OCR、図面解析 | 正解データ（アノテーション）の準備が重い。顧客側のデータ準備負荷を明示 |
| **数値予測系** | 回帰、分類、時系列予測 | 定量評価が主。精度指標の合意が重要 |
| **混合** | 複数のAI機能を組み合わせ | 検証項目ごとにタイプを判定し、それぞれの作業分担を整理 |

判定結果に応じて、Slide 4（作業分担）の「準備」カテゴリの内容を調整する。

### Step 2: 6枚分の情報抽出

各スライドに必要な情報をインプットから抽出する：

| スライド | 必要情報 | 抽出元の優先順位 |
|---------|---------|----------------|
| Slide 1（SoW記載事項） | SoW全体像（固定テンプレート） | デフォルト値（カスタマイズは備考のみ） |
| Slide 2（検証項目） | AI機能名、対象データ、データ量、絞り込み条件 | 提案資料 > 議事録 > チェックリスト |
| Slide 3（評価内容） | 評価方法、定量指標、定性評価の方法 | 提案資料 > 議事録 |
| Slide 4（作業分担） | 準備・評価・報告の双方タスク、参加者、ツール | 提案資料 > 議事録 |
| Slide 5（検証サイクル） | 報告頻度、連絡ツール、成果物 | 提案資料 > 議事録 |
| Slide 6（工程表） | 期間、Phase構成、マイルストーン | 提案資料 > 議事録 |

### Step 3: 不足情報の特定

抽出できなかった情報を洗い出し、以下の処理を行う：

- **プレースホルダーの挿入**: `[要確認]`、`[XX様]`、`[ツール名]` 等
- **デフォルト値の適用**: 上記各スライドの「抽出ルール」に記載されたデフォルトを使用
- **不足情報サマリーの生成**: ドラフト末尾に「確認が必要な項目」として一覧化

### Step 4: ドラフト生成

以下の出力フォーマットに従い、Markdownファイルを生成する。

### Step 5: 品質チェック

- [ ] スコープアウト対象が明示されているか（Slide 2 の「次フェイズ以降」行）

#### 合意形成チェック

## 処理ルール（厳守事項）

1. **6枚固定構成**: 必ず6枚すべてを生成する。情報が不足していてもプレースホルダーを入れて構成を崩さない。
2. **合意形成の3箇所**: Slide 1（合意前提の明示）、Slide 4（顧客の関与依頼）、Slide 6（変動可能性の但し書き）の3箇所の冒頭メッセージは**必ず含める**。
3. **検証項目の一貫性**: Slide 2（検証項目）の各No.がSlide 3（評価内容）・Slide 6（工程表）に必ず反映されていること。
4. **「次フェイズ以降」行の必須化**: Slide 2・Slide 3 の最終行に「本 Phase では実施せず、次フェイズ以降での実施を検討」行を必ず含める。
5. **定性評価と顧客協力の連動**: Slide 3 で定性評価がある場合、Slide 4（作業分担）の「評価」カテゴリに顧客の協力作業が必ず記載されていること。
6. **プレースホルダーの可視化**: 不明情報は具体的なプレースホルダー（`[XX様]`、`[ツール名]`、`[約XX件]` 等）で明示し、末尾の「確認が必要な項目」に集約する。
7. **プロジェクトタイプに応じた調整**: Step 1 で判定したタイプ（LLM系/画像系/数値予測系/混合）に応じて、Slide 4（作業分担）の「準備」カテゴリの内容を適切に調整する。

## 関連スキルとの連携

| スキル | 関係 | 連携内容 |
|--------|------|---------|
| **sow-generator** | 兄弟スキル | 要件定義PJ向けSoW。PoCフェーズには本スキル、要件定義フェーズには sow-generator を使用 |
| **project-definition-checker** | 上流（インプット） | チェックリスト診断結果をインプットとして使用 |
| **discussion-slide-draft** | 並行 | 議論用ドラフトで合意した内容をSoWに反映 |
| **consulting-slide** | 下流（アウトプット） | 本スキルのドラフトを渡してスライド化を依頼 |
| **post-meeting-pipeline** | 上流 | 打ち合わせ後処理パイプラインの中でSoW更新が必要な場合に呼び出される |

### 運用フロー（全体像）

```
提案フェーズ完了 → PoCフェーズ開始
    ↓
提案資料・議事録・チェックリストを入力
    ↓
★ sow-generator-poc → PoC SoWドラフト作成（本スキル）
    ↓
ユーザー確認・修正
    ↓
consulting-slide → .pptx 生成
    ↓
顧客提出・合意
    ↓
PoC実行（SoWに基づく）
    ↓
PoC完了 → Phase 2 判断
    ↓
（Phase 2 に進む場合）sow-generator → 要件定義SoW作成
```
